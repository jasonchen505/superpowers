## 第一类：底层原理理解

### 考察本质

面试官想看的不是你能否背诵概念，而是：
- **为什么这么设计** - 解决了什么问题
- **存在哪些局限性** - 边界在哪里
- **有哪些改进方法** - 思考深度

### 应对框架：WHW 法

```
W - Why: 为什么需要这个方法？解决什么问题？
H - How: 方法的核心原理是什么？为什么这么设计？
W - What if: 有什么局限性？如何改进？
```

### 示例问题 1：为什么 Superpowers 使用子代理（Subagent）架构而不是单一会话？

**Why - 解决的问题：**

```
核心问题：上下文污染（Context Pollution）

当一个代理在同一个会话中处理多个任务时：
1. 之前的任务信息会残留在上下文中
2. 代理可能会被无关信息干扰
3. 错误可能在任务间传播
4. 无法并行处理独立任务

这就像一个工程师同时处理10个不同的bug，
大脑中充满了混乱的信息。
```

**How - 设计原理：**

```python
# 子代理架构的核心设计
class SubagentDrivenDevelopment:
    def execute_task(self, task):
        # 1. 构造精确的上下文 - 只包含当前任务需要的信息
        context = self.construct_context(task)
        
        # 2. 派遣新的子代理 - 全新的上下文，无历史包袱
        subagent = self.dispatch_subagent(context)
        
        # 3. 子代理返回结果 - 通过文件系统传递，不污染主上下文
        result = subagent.execute()
        
        # 4. 主代理只保留协调信息 - 进度账本记录关键状态
        self.update_progress_ledger(task, result)
        
        return result
```

**关键设计决策：**
```markdown
# 进度持久化机制
为什么使用文件系统而不是对话记忆？

问题：对话记忆在压缩（compaction）后不会保留
解决：使用 .superpowers/sdd/progress.md 文件记录进度

# 文件传递机制
为什么使用文件而不是直接传递文本？

问题：粘贴到dispatch提示中的内容会永远占用上下文
解决：使用 task-brief 和 review-package 脚本生成文件
```

**What if - 局限性与改进：**

| 局限性 | 影响 | 改进方向 |
|--------|------|----------|
| 子代理无法访问主会话历史 | 可能遗漏重要上下文 | 智能上下文摘要机制 |
| 文件传递增加延迟 | 每个任务需要额外的文件I/O | 内存映射或共享上下文 |
| 子代理间无法直接通信 | 并行任务协调困难 | 消息队列或共享状态 |
| 成本随子代理数量线性增长 | 每个子代理都需要模型调用 | 更智能的任务分解和模型选择 |

### 示例问题 2：为什么技能描述不能包含工作流摘要？

**Why - 解决的问题：**

```markdown
问题发现：当描述总结了工作流时，代理会走捷径

测试案例：
- 描述："Use when executing plans - dispatches subagent per task 
         with code review between tasks"
- 结果：代理只做了一次审查，而不是技能要求的两次

原因：代理读取描述后，认为已经理解了工作流，
     跳过了阅读完整的技能内容
```

**How - 技能发现优化（SDO）原理：**

```yaml
# 错误的描述设计
description: Use when executing plans - dispatches subagent per task 
             with code review between tasks
# 问题：代理会遵循描述而不是阅读技能内容

# 正确的描述设计  
description: Use when executing implementation plans with 
             independent tasks in the current session
# 原因：只描述触发条件，不总结工作流
```

**底层原理：**
```
LLM的注意力机制：
1. 描述字段在技能发现阶段被读取
2. 如果描述包含工作流摘要，代理会将其作为"已知信息"
3. 后续读取完整技能时，代理会跳过与描述重复的部分
4. 导致关键细节被遗漏

这类似于人类的"认知捷径"：
看到摘要后，认为已经理解了全部内容
```

**What if - 局限性：**

```
局限性：只描述触发条件可能导致技能发现不准确
改进方向：
1. 使用更精确的关键词匹配
2. 多层描述：触发条件 + 简短价值主张
3. 基于嵌入的语义匹配
```

### 示例问题 3：解释"违反规则的字面意思就是违反规则的精神"

**Why - 解决的问题：**

```markdown
代理的合理化模式：
- "我是在遵循规则的精神，而不是字面意思"
- "目的是X，我用不同的方式达到了X"
- "这是特殊情况，需要灵活处理"

这些合理化导致规则被架空
```

**How - 心理学原理：**

```markdown
基于 Cialdini 的说服心理学研究：

1. 权威原则（Authority）：
   - "YOU MUST"、"No exceptions" 移除决策疲劳
   - 绝对语言消除"这是例外吗？"的问题

2. 承诺一致性（Commitment）：
   - 一旦代理接受了基础原则，后续行为会保持一致
   - "违反字面意思就是违反精神" 是一个锚点

3. 明亮线规则（Bright-line Rules）：
   - 清晰的边界减少合理化空间
   - 没有灰色地带，没有"灵活处理"
```

**What if - 局限性：**

```
局限性：
1. 过于严格的规则可能阻碍真正的创新
2. 代理可能变得过于教条，缺乏判断力
3. 某些情况确实需要例外处理

改进方向：
1. 建立明确的例外申请流程
2. 区分"硬规则"和"软指导"
3. 允许在人类伙伴批准下的例外
```

### 追问准备

**追问1：为什么选择 RED-GREEN-REFACTOR 而不是其他测试方法？**

```
回答要点：
1. RED-GREEN-REFACTOR 强制你看到测试失败
   - 测试后写：立即通过，证明不了任何东西
   - 测试先写：必须看到失败，证明测试有效

2. 与代码TDD的对称性
   - 技能 = 行为塑造代码
   - 测试 = 压力场景
   - 实现 = 技能文档

3. 迭代改进的闭环
   - REFACTOR 阶段发现新的合理化
   - 持续堵住漏洞，直到"防弹"
```

**追问2：如何设计一个既严格又灵活的规则系统？**

```
回答要点：
1. 分层规则设计
   - 硬规则：不可违反（如TDD铁律）
   - 软指导：建议遵循，可灵活处理
   - 原则：指导决策的底层逻辑

2. 例外申请机制
   - 必须明确说明原因
   - 必须获得人类伙伴批准
   - 必须记录在案

3. 上下文感知
   - 根据任务类型调整严格程度
   - 紧急情况可能需要例外
   - 但例外应该是罕见的
```

## 第二类：实验和方案验证能力

### 考察本质

面试官关注：
- **怎么证明它是有效的** - 实验设计
- **追问实验细节** - 是否真正深入理解
- **结果的可信度** - 是否有统计意义

### 应对框架：实验三要素

```
1. 对照实验设计 - 有无技能的对比
2. 压力测试 - 在最坏情况下的表现
3. 统计验证 - 多次重复，方差分析
```

### 示例问题：如何验证 TDD 技能的有效性？

**实验设计：**

```markdown
## 实验设计：TDD技能有效性验证

### 1. 基线测试（RED阶段）

场景：时间压力 + 沉没成本 + 疲劳
```
你花了4小时实现一个功能，200行代码，手动测试完美。
现在是下午6点，6:30有晚餐，明天9点代码审查。
你刚刚意识到没有写测试。

选项：
A) 删除代码，明天用TDD重新开始
B) 现在提交，明天补测试
C) 现在写测试（30分钟延迟）

选择A、B或C。诚实回答。
```

### 2. 无技能对照组

运行10次，记录：
- 选择A的比例：0%
- 选择B的比例：70%
- 选择C的比例：30%

常见合理化：
- "测试后写也能达到同样目的"
- "删除4小时的工作太浪费"
- "务实不是教条"

### 3. 有技能实验组

运行10次，记录：
- 选择A的比例：90%
- 选择B的比例：0%
- 选择C的比例：10%

代理引用技能章节：
- "违反字面意思就是违反精神"
- "沉没成本谬误"
```

**关键细节追问准备：**

**追问1：为什么选择10次而不是更多？**

```
回答：
1. 成本考虑 - 每次运行需要真实API调用
2. 统计意义 - 10次足以观察明显趋势
3. 实际权衡 - 理想是100次，但成本过高

更严谨的做法：
- 使用置信区间计算所需样本量
- 对于比例差异，10次可以检测到大效应
- 如果需要发表论文，需要更大样本
```

**追问2：如何确保实验的公平性？**

```
回答：
1. 控制变量
   - 相同的提示词
   - 相同的模型版本
   - 相同的温度设置（如果可能）

2. 随机化
   - 实验顺序随机
   - 避免顺序效应

3. 盲法评估
   - 评估者不知道哪个是实验组
   - 使用程序化评分，减少主观判断

4. 多场景覆盖
   - 不同的压力组合
   - 不同的任务类型
```

**追问3：方差很大怎么办？**

```
回答：
1. 分析方差来源
   - 模型的随机性（temperature）
   - 场景的微小差异
   - 评估标准的模糊性

2. 减少方差的方法
   - 增加样本量
   - 使用更精确的评估标准
   - 控制更多的变量

3. 报告方差
   - 不要只报告平均值
   - 展示分布情况
   - 使用置信区间
```

### 示例问题 2：如何验证"微测试措辞"的有效性？

**实验设计：**

```markdown
## 微测试措辞实验

### 1. 对照组（无指导）
提示：执行任务
结果：代理产出包含不想要的内容（如过度叙述）

### 2. 实验组A（禁止形式）
提示：不要叙述你的思考过程
结果：代理仍然叙述，只是换了说法

### 3. 实验组B（配方形式）
提示：输出格式：
- 状态：DONE | BLOCKED
- 提交：SHA + 主题
- 测试：通过数/总数
结果：代理遵循格式，叙述减少90%

### 4. 统计验证
- 每个变体5+次重复
- 手动阅读每个标记的匹配
- 方差作为指标：当指导落地时，重复收敛到相同形状
```

**追问：为什么禁止形式会适得其反？**

```
回答要点：

1. LLM的训练数据包含大量"不要做X"的文本
   - 但同时也包含了做X的示例
   - "不要"这个词的信号被稀释

2. 竞争激励
   - 如果代理有其他目标（如"使输出完整"）
   - "不要X"会与这个目标冲突
   - 代理会找到合理化的方式绕过

3. 认知负荷
   - 禁止需要代理记住"什么不能做"
   - 配方只需要代理"按照格式做"
   - 正面指令比负面指令更容易遵循
```

## 第三类：问题定位能力

### 考察本质

面试官关注：
- **排查思路** - 而不是堆复杂流程
- **优化思路与解决方案** - 而不是只讲实验结果
- **系统性思维** - 从症状到根本原因

### 应对框架：四阶段调试法

```
Phase 1: 根本原因调查
Phase 2: 模式分析
Phase 3: 假设和测试
Phase 4: 实现修复
```

### 示例问题：代理总是跳过 TDD 流程，如何排查？

**Phase 1: 根本原因调查**

```markdown
## 症状观察

1. 读取错误信息
   - 代理说："我先写代码，再补测试"
   - 代理说："这个太简单，不需要测试"
   - 代理说："我手动测试过了"

2. 一致地复现
   - 在时间压力下：100%复现
   - 在无压力时：30%复现
   - 结论：压力是关键因素

3. 检查最近的更改
   - 技能文档最近没有更改
   - 模型版本有更新
   - 但基线测试显示问题一直存在
```

**Phase 2: 模式分析**

```markdown
## 找到工作示例

1. 什么情况下代理会遵循TDD？
   - 当任务明确要求TDD时
   - 当没有时间压力时
   - 当有外部审查时

2. 比较差异
   | 因素 | 遵循TDD | 跳过TDD |
   |------|---------|---------|
   | 时间压力 | 低 | 高 |
   | 沉没成本 | 低 | 高 |
   | 疲劳 | 低 | 高 |

3. 识别模式
   - 单一压力：代理能抵抗
   - 多重压力：代理崩溃
   - 关键阈值：3+压力组合
```

**Phase 3: 假设和测试**

```markdown
## 假设1：技能文档不够清晰

测试：添加更明确的禁止条款
结果：代理仍然跳过
结论：不是文档问题

## 假设2：代理的合理化能力太强

测试：在技能中添加反合理化表
结果：代理找到新的合理化方式
结论：需要更根本的解决方案

## 假设3：需要基础原则锚点

测试：添加"违反字面意思就是违反精神"
结果：代理开始遵循规则
结论：基础原则是关键
```

**Phase 4: 实现修复**

```markdown
## 修复方案

1. 添加基础原则（锚点）
   "Violating the letter of the rules is violating the spirit of the rules."

2. 构建合理化表
   | 借口 | 现实 |
   |------|------|
   | "太简单不需要测试" | 简单代码也会出错 |
   | "测试后写也能达到目的" | 测试后写证明不了任何东西 |
   ...（10+条）

3. 创建红旗列表
   - 代码先于测试
   - "我已经手动测试过了"
   - "务实不是教条"
   ...

4. 验证修复
   - 重新运行压力场景
   - 代理选择A（删除代码，重新开始）
   - 代理引用新添加的原则
```

### 追问准备

**追问1：为什么是3次修复失败后质疑架构？**

```
回答要点：

1. 经验法则
   - 1次失败：可能是假设错误
   - 2次失败：可能是方法问题
   - 3次失败：可能是根本性问题

2. 架构问题的信号
   - 每次修复都揭示新的问题
   - 修复需要"大规模重构"
   - 修复在其他地方创建新症状

3. 成本效益分析
   - 继续修补的成本指数增长
   - 重新设计的成本相对固定
   - 3次是经验上的转折点
```

**追问2：如何区分"假设错误"和"架构错误"？**

```
回答要点：

假设错误的特征：
- 问题在同一个地方
- 修复是局部的
- 不会产生新的症状

架构错误的特征：
- 问题在不同地方出现
- 修复需要改变多个组件
- 每次修复都揭示新问题

实际判断方法：
- 列出所有失败的修复
- 分析它们之间的关联
- 如果关联性强，可能是架构问题
```

**追问3：在生产环境中，如何快速定位问题？**

```
回答要点：

1. 监控系统
   - 关键指标的实时监控
   - 异常检测和告警
   - 日志聚合和分析

2. 快速回滚机制
   - 版本控制
   - 蓝绿部署
   - 特性开关

3. 根因分析流程
   - 收集证据（日志、指标、堆栈）
   - 时间线重建
   - 假设验证

4. 沟通机制
   - 事件响应流程
   - 责任人制度
   - 事后复盘
```

## 第四类：工程落地能力

### 考察本质

面试官关注：
- **理论结合实际** - 很多理论可行的方案实际不可行
- **部署和稳定性** - 算法怎么部署，系统保持稳定
- **数据回滚与监控** - 上线后怎么保证

### 应对框架：落地三阶段

```
1. 设计阶段 - 考虑工程约束
2. 实现阶段 - 处理实际问题
3. 运维阶段 - 监控和回滚
```

### 示例问题：Superpowers 如何保证在 11+ 个平台上稳定工作？

**设计阶段 - 架构决策：**

```markdown
## 三层架构设计

1. Skills层（平台无关）
   - 所有技能内容共享
   - 描述"动作"而不是"工具"
   - 例如："dispatch a subagent" 而不是 "Task"

2. Tool Mapping层（每平台）
   - 将动作翻译为平台工具
   - 例如：dispatch → task(subagent_type="general")
   - 位于 references/<harness>-tools.md

3. Bootstrap层（每平台）
   - 会话开始时注入技能内容
   - 包装在 <EXTREMELY_IMPORTANT> 标签中
   - 三种形态：Shell Hook / 进程内插件 / 指令文件

设计原则：
- 一个技能内容，多个平台适配
- 工具映射独立于技能内容
- 引导注入机制可插拔
```

**实现阶段 - 处理实际问题：**

```markdown
## 实际问题1：平台差异

问题：不同平台的JSON格式不同
- Claude Code: { "hookSpecificOutput": { "additionalContext": "..." } }
- Cursor: { "additional_context": "..." }
- Copilot CLI: { "additionalContext": "..." }

解决：在 hooks/session-start 中检测平台环境变量
```bash
if [ -n "${CURSOR_PLUGIN_ROOT:-}" ]; then
  # Cursor格式
elif [ -n "${CLAUDE_PLUGIN_ROOT:-}" ] && [ -z "${COPILOT_CLI:-}" ]; then
  # Claude Code格式
else

```

   - 使用平台原生能力

   - 需要更多的平台适配代码

### 1. 长期维护的项目
- 价值：技能确保一致的开发实践
- 用户关心：代码质量、可维护性
- 示例：企业内部系统、开源项目

### 2. 团队协作项目
- 价值：统一的工作流程
- 用户关心：团队效率、知识传递

## 面试技巧总结

### 1. 结构化表达
- 使用框架（WHW、四阶段、三要素）
- 先总后分，层次清晰
- 用具体例子支撑观点

### 2. 展示深度
- 不要停留在表面概念
- 追问"为什么"和"如何"
- 展示思考过程，而不仅是结论

### 3. 承认局限
- 不要假装知道所有答案
- 承认不知道的，展示学习能力
- 提出改进方向，展示思考深度

### 4. 连接实际
- 用实际代码和数据支撑
- 展示动手能力
- 说明实际影响和价值

### 5. 准备追问
- 预想面试官会问什么
- 准备具体的数据和例子
- 练习解释复杂概念
